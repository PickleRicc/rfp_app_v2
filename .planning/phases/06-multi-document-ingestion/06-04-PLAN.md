<objective>
Build the solicitation detail UI showing reconciliation results, amendment chain views, template field mappings, and version lineage. This is the user-facing culmination of all Phase 6 work — where users see the "unified, reconciled requirement set."

Purpose: Users need to see what changed between document versions, understand which amendments/Q&A modified which requirements, review fillable templates, and have confidence that the system produced the correct "current truth" from the solicitation package.

Output: Solicitation detail page with tabbed views, reconciliation display with strikethrough and undo, amendment chain visualization, template fields panel, and solicitations list page.
</objective>

**app/components/solicitation/ReconciliationView.tsx:**

Create the reconciliation display component implementing user decisions for how superseded text is shown.

Props: `{ reconciliations: DocumentReconciliation[], documents: SolicitationDocument[], onToggle: (reconciliationId: string, isActive: boolean) => void }`

Implementation:
- Group reconciliations by target_document (the document being modified)
- For each group, show the document name as a section header
- For each reconciliation record:
  - Show source badge: "Amendment {amendment_number}" (amber) or "Q&A Response" (green) — per user decision: "Q&A responses tagged as 'Q&A-sourced'"
  - Show change_type badge (e.g., "Supersedes Section", "Modifies Page Limit", "Modifies Eval Criteria")
  - Section reference (e.g., "Section L.4.2", "Question 15")
  - **Original text**: Show with inline strikethrough styling (line-through text-muted-foreground) when is_active=true. Per user decision: "Inline strikethrough for superseded text"
  - **Replacement text**: Show below the strikethrough with a left border accent (border-l-4 border-amber-400 for amendments, border-green-400 for Q&A)
  - **Link to source**: Clickable link showing which amendment/Q&A made this change. Per user decision: "with a link/tag to the amendment that replaced it"
  - **Undo/Restore button**: Toggle button that calls onToggle(id, !is_active). When is_active=false, show the original text without strikethrough and mark the reconciliation as "Overridden by user". Per user decision: "Users can override reconciliation with undo/restore action"
- When no reconciliations exist, show empty state: "No reconciliation changes detected"

**app/components/solicitation/AmendmentChainView.tsx:**

Create the amendment chain visualization per user decision: "Chain view for multi-amendment evolution: Original -> Amend 1 -> Amend 2, with only the latest version active"

Props: `{ documents: SolicitationDocument[], reconciliations: DocumentReconciliation[] }`

Implementation:
- Build the amendment chain from documents: base_rfp -> amendments sorted by amendment_number
- Render as a horizontal timeline or vertical chain:
  - Each node shows: document name, type badge, effective_date
  - Active node (latest) gets a green border/highlight, "ACTIVE" badge
  - Superseded nodes get a muted/faded appearance with strikethrough on the name
  - Arrows between nodes show the progression
- Below the chain, show summary stats: total changes per amendment, change types
- If only one document (no amendments), show a simple "Base Document — No Amendments" view
- This component is embedded in the Documents tab of the detail page, above the classification table
