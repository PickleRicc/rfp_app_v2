    **Migration SQL** — Create `supabase_tables/supabase-compliance-extraction-migration.sql`:

    Create one table: `compliance_extractions` — stores individual extracted data points.

    Schema:
    ```
    compliance_extractions:
      id                  UUID PK DEFAULT gen_random_uuid()
      solicitation_id     UUID NOT NULL REFERENCES solicitations(id) ON DELETE CASCADE
      source_document_id  UUID REFERENCES solicitation_documents(id) ON DELETE SET NULL
      category            TEXT NOT NULL CHECK (category IN ('section_l', 'section_m', 'admin_data', 'rating_scales'))
      field_name          TEXT NOT NULL  -- e.g., "volume_structure", "page_limit_technical", "evaluation_methodology"
      field_label         TEXT NOT NULL  -- Human-readable label for display
      field_value         JSONB         -- Flexible value storage (string, number, array, nested object)
      confidence          TEXT NOT NULL DEFAULT 'medium' CHECK (confidence IN ('high', 'medium', 'low'))
      is_user_override    BOOLEAN DEFAULT false
      original_ai_value   JSONB         -- Stores AI value when user overrides (for revert)
      amendment_source_id UUID REFERENCES solicitation_documents(id) ON DELETE SET NULL  -- Which amendment changed this
      amendment_note      TEXT          -- e.g., "Updated from Amendment 0003"
      extraction_status   TEXT NOT NULL DEFAULT 'pending' CHECK (extraction_status IN ('pending', 'extracting', 'completed', 'failed'))
      error_message       TEXT
      created_at          TIMESTAMPTZ DEFAULT NOW()
      updated_at          TIMESTAMPTZ DEFAULT NOW()
    ```

    Add indexes:
    - `idx_compliance_extractions_solicitation` on (solicitation_id)
    - `idx_compliance_extractions_category` on (solicitation_id, category)
    - `idx_compliance_extractions_source_doc` on (source_document_id)

    Enable RLS with permissive authenticated-user policy (same pattern as Phase 6 migration).
    Use IF NOT EXISTS throughout. Add COMMENT ON TABLE.

    **Extraction Engine** — Create `lib/ingestion/compliance-extractor.ts`:

    Import `anthropic, MODEL` from `@/lib/anthropic/client`. Import the extractJSON helper pattern from solicitation-classifier.ts (copy the function locally or import if available).

    Export 4 async functions, each calling Anthropic with the reconciled document text and returning typed extraction results:

    1. `extractSectionL(documentTexts: string[]): Promise<SectionLExtraction[]>`
       - Concatenate relevant document texts (base_rfp + active reconciliation replacements, max 30000 chars)
       - Prompt Claude to extract: volume structure (names + page limits + format rules per volume), required attachments, required forms/certifications, formatting rules (font, size, margins, spacing)
       - Each extracted field becomes a ComplianceExtraction-ready object with field_name, field_label, field_value (JSONB), and confidence
       - Return array of extraction objects (one per distinct field: "volume_structure", "required_attachments", "required_forms", "formatting_rules")

    2. `extractSectionM(documentTexts: string[]): Promise<SectionMExtraction[]>`
       - Prompt Claude to extract: evaluation factors with subfactors and weight descriptions, evaluation methodology (LPTA vs best value vs tradeoff), experience thresholds, key personnel requirements
       - Return extraction objects: "evaluation_factors", "evaluation_methodology", "experience_thresholds", "key_personnel_requirements"

    3. `extractAdminData(documentTexts: string[]): Promise<AdminDataExtraction[]>`
       - Prompt Claude to extract: NAICS code, size standard, set-aside designation, contract type, period of performance, CLIN structure
       - Return extraction objects: "naics_code", "size_standard", "set_aside_designation", "contract_type", "period_of_performance", "clin_structure"

    4. `extractRatingScales(documentTexts: string[]): Promise<RatingScaleExtraction[]>`
       - Prompt Claude to extract: rating scale definitions per factor, relative factor weightings/importance descriptions
       - Return extraction objects: "rating_scales", "factor_weightings"

    Each function should:
    - Use `anthropic.messages.create()` with `MODEL`, max_tokens 4096, temperature 0
    - Include a system prompt establishing the role as a government proposal compliance analyst
    - Request JSON response with specific schema per extraction type
    - Use extractJSON to parse response, with try/catch returning empty array on parse failure
    - Set confidence per field: 'high' when Claude explicitly states data was found, 'medium' when inferred, 'low' when uncertain or partial
    - Include source_text_snippet in response for reference (first 200 chars of relevant source text)

    Each extraction result object shape:
    ```typescript
    interface ExtractionResult {
      field_name: string;
      field_label: string;
      field_value: unknown; // JSONB-compatible
      confidence: 'high' | 'medium' | 'low';
    }
    ```

    **Inngest Function** — Create `lib/inngest/functions/solicitation-compliance-extractor.ts`:

    Export `solicitationComplianceExtractor` — Inngest function triggered by `solicitation.reconciliation.complete` event.

    Event payload: `{ solicitationId: string }`

    Use the same Inngest step pattern as solicitation-reconciler.ts (step.run() for each step, processing_logs for observability).

    **6-Step Pipeline:**

    1. `fetch-documents` — Load solicitation + all reconciled documents. Build concatenated text from base_rfp and soo_sow_pws documents (applying active reconciliation replacements). Set solicitation status to 'extracting' (add 'extracting' to the SolicitationStatus type and DB CHECK constraint — or keep as 'ready' and use extraction_status on the compliance_extractions table instead; prefer the latter to avoid modifying Phase 6 schema). Insert initial 'pending' compliance_extractions rows for all 4 categories (one row per expected field per category).

    2. `extract-section-l` — Call `extractSectionL()` with document texts. For each result, upsert to compliance_extractions (match on solicitation_id + field_name, update field_value + confidence + extraction_status='completed'). If extraction fails: set category rows to extraction_status='failed' with error_message. Log to processing_logs.

    3. `extract-section-m` — Call `extractSectionM()`. Same upsert pattern. Log to processing_logs.

    4. `extract-admin-data` — Call `extractAdminData()`. Same upsert pattern. Log to processing_logs.

    5. `extract-rating-scales` — Call `extractRatingScales()`. Same upsert pattern. Log to processing_logs.

    6. `finalize` — Count completed vs failed extractions. Log summary to processing_logs. Each section updates independently (per user decision — "Each section updates independently as extraction completes").

    Error isolation pattern: Each step catches its own errors and continues. A failure in Section L extraction does not prevent Section M extraction from running.

    **Event trigger addition** — Modify `lib/inngest/functions/solicitation-reconciler.ts`:
    In the `finalize` step, after setting solicitation status to 'ready', add:
    ```typescript
    await inngest.send({
      name: 'solicitation.reconciliation.complete',
      data: { solicitationId }
    });
    ```
    Only send the event when status is set to 'ready' (not 'failed').

    **Register function** — Add `solicitationComplianceExtractor` import and entry to the functions array in `app/api/inngest/route.ts`.

    Create `app/api/solicitations/[id]/compliance/route.ts` with two handlers:

    **GET /api/solicitations/[id]/compliance**
    - Auth: `requireStaffOrResponse()` + X-Company-Id header
    - Verify solicitation ownership (company_id match)
    - Query compliance_extractions WHERE solicitation_id = id
    - Group results by category into `ComplianceExtractionsByCategory` shape
    - Return `{ extractions: { section_l: [...], section_m: [...], admin_data: [...], rating_scales: [...] } }`
    - Include extraction_status per row so UI knows which sections are still loading

    **PATCH /api/solicitations/[id]/compliance**
    - Auth: same as GET
    - Body options:
      1. **Edit a value**: `{ extraction_id, field_value }` — Updates field_value, sets is_user_override=true, saves original AI value to original_ai_value (only on first override — don't overwrite if already set), updates updated_at
      2. **Re-extract a category**: `{ category, action: 're-extract' }` — Resets all extractions for that category to extraction_status='pending', clears error_message. Send `solicitation.compliance.re-extract` Inngest event with `{ solicitationId, category }` to trigger re-extraction for just that category.
      3. **Revert user override**: `{ extraction_id, action: 'revert' }` — Restores original_ai_value to field_value, sets is_user_override=false
    - Validate extraction_id belongs to the solicitation
    - Return updated extraction record(s)
